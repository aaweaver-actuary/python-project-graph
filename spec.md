# Code Dependency Graph Explorer — V1 Product & Technical Specification

## Purpose

Build an interactive web application for exploring Python repository dependencies at the symbol level.

The system consumes graph JSON generated by an existing Rust analysis application and renders an explorable, filterable, draggable dependency graph.

The primary goal is **faster structural understanding of unfamiliar repositories** than traditional IDE navigation.

This is **not** a general whiteboard tool.

This is a **typed dependency exploration interface** for code structure and flow.

---

## Primary Goals

### 1. Repository dependency exploration
Allow a developer to quickly answer:

- What depends on this symbol?
- What does this symbol depend on?
- Where does data flow downstream?
- Which modules act as bottlenecks?
- Where are utility/helper layers concentrated?

### 2. Visual structural understanding
Provide an immediate visual representation of:

- modules
- classes
- functions
- methods
- constants
- imports

and their dependency relationships.

### 3. Interactive investigation
Support direct manipulation and exploration:

- drag nodes
- zoom
- pan
- select
- inspect
- isolate neighborhoods
- expand upstream/downstream paths

### 4. Future extensibility
Architecture should support future V2 features:

- duplicate helper detection
- semantic similarity edges
- developer ownership overlays
- change history overlays
- git blame integration
- architectural linting

---

## Non-Goals (V1)

The following are explicitly out of scope for V1:

- editing source code
- bidirectional IDE sync
- live repository monitoring
- collaborative multi-user editing
- AI-generated architecture suggestions
- duplicate function similarity scoring
- full AST inspection UI

## Input Data Contract

The Rust application is the source of truth.

It emits a list of typed nodes and directed edges.

### Node schema

```json
{
  "id": "module.utils.parse_config",
  "kind": "function",
  "name": "parse_config",
  "module": "module.utils",
  "file_path": "src/module/utils.py",
  "line_start": 42,
  "line_end": 68,
  "docstring": "Parse configuration object.",
  "topological_rank": 3
}
```

### Allowed node kinds

```text
module
class
method
function
import
constant
variable
package
```

### Edge schema

```json
{
  "source": "module.utils.parse_config",
  "target": "module.pipeline.run_model",
  "kind": "dependency"
}
```

### Allowed edge kinds

```text
dependency
imports
inherits
calls
contains
references
```

## Functional Requirements

## FR-1 Graph Rendering

Render all nodes and edges from input JSON.

Requirements:

- directed graph
- arrowheads on edges
- zoomable canvas
- pannable infinite workspace
- draggable nodes
- minimap

Preferred library:

```text
React Flow
```

---

## FR-2 Initial Layout

Initial layout must be auto-generated.

Use layered DAG layout.

Preferred flow direction:

```text
left -> right
```

Where:

- foundational imports/modules on left
- downstream consumers on right

Fallback:

```text
top -> bottom
```

if horizontal layout becomes too wide.

Use topological rank if available from Rust.

Otherwise compute client-side.

---

## FR-3 Node Styling by Type

Each node type must have a visually distinct style.

Example mapping:

```text
module      -> rectangular large card
class       -> rounded rectangle
method      -> compact rounded rectangle
function    -> rectangle
constant    -> pill
import      -> muted small node
```

Distinction should rely on:

- shape
- border style
- icon
- label prefix
- color token (theme-safe)

Do not rely solely on color.

---

## FR-4 Node Interaction

On click:

- highlight node
- highlight immediate inbound edges
- highlight immediate outbound edges
- open detail panel

On hover:

- show quick metadata tooltip

On double click:

- focus neighborhood view

---

## FR-5 Detail Panel

Right-side detail panel.

Display:

- symbol name
- symbol type
- module path
- file path
- line range
- inbound dependency count
- outbound dependency count
- immediate parents
- immediate children

Optional if available:

- docstring
- signature
- ownership metadata

---

## FR-6 Filtering

Required filters:

### by node type
Multi-select:

- modules
- classes
- methods
- functions
- constants
- imports

### by module/package
Text search + dropdown

### by file path
substring match

### isolated node suppression
Hide disconnected nodes

---

## FR-7 Neighborhood Focus

Required exploration actions:

### upstream (1 hop)
Show direct dependencies

### downstream (1 hop)
Show direct consumers

### expand recursively
User-selectable depth:

```text
1
2
3
all
```

### isolate path
Show only selected node and reachable neighbors

This is critical.

This is the core value proposition.

---

## FR-8 Search

Search box must support:

- symbol name
- partial symbol name
- module path
- file path

Result action:

- zoom to node
- highlight node
- open detail panel

---

## FR-9 Manual Layout Override

User may drag nodes.

Dragged positions should persist during session.

Optional persistence:

```text
localStorage
```

for V1.1

## Performance Requirements

V1 target:

```text
500–2000 nodes
2000–10000 edges
```

Interactive latency target:

```text
<100ms
```

for:

- click
- pan
- zoom
- filter updates

Initial render:

```text
<2 seconds
```

for 1000 nodes.

---

## UX Success Criteria

A developer should be able to answer within 10 seconds:

- what depends on X?
- what does X depend on?
- what module is most central?
- where does this utility function feed downstream?

If not, V1 has failed regardless of visual quality.

## Core Product Principle

Optimize for:

```text
structural comprehension speed
```

not aesthetic diagramming.

The graph must answer questions faster than:

- grep
- IDE symbol search
- file tree navigation
- import tracing

This should remain the guiding principle for all implementation decisions.
